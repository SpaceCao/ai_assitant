# HiAgent意图识别最终交付说明

本文用于汇总本次 HiAgent 意图识别方案的全部交付物，并给出推荐使用顺序。

## 1. 交付目标

本次交付只覆盖 HiAgent 平台中的 `意图识别` 能力建设，目标是先完成一条稳定、可调试、可回归的最小识别链路。

不包含：

- 插件接入
- 子智能体调用
- 槽位补全执行
- 业务接口调用

## 2. 最终流程

推荐在平台上按以下流程落地：

```text
Start
  -> Python（intent_mapping_workflow，action=get_enabled_intents）
  -> 大模型（意图识别与结构化输出）
  -> 选择器
     -> need_clarify == false -> End
     -> need_clarify == true -> 消息（澄清） -> End
```

说明：

- `Start` 提供用户输入 `query`
- `Python` 前置节点负责输出 `intents` 和 `default_clarify_question`
- `大模型` 只根据前置节点输出做结构化识别，并输出下游可直接消费的 `execution_plan`
- `选择器` 负责按 `need_clarify` 分流
- `消息` 仅用于澄清
- `execution_plan` 采用递归树结构，同时承载识别结果、串行/并行关系、执行顺序和多层嵌套

## 8. 验收口径

验收时重点看：

- Python 前置节点输出的 `intents` 是否完整
- 高频样例识别是否稳定
- 边界样例是否明显误判
- 多意图输入是否能稳定输出主意图，`execution_plan.root` 是否合理
- 模糊输入是否优先落到 `need_clarify=true`
- `reason_text` 是否有解释价值

建议目标：

- 主意图识别准确率 >= 90%
- 模糊输入兜底稳定
- 边界样例误判率可控

## 9. 后续迭代建议

只有当意图识别稳定后，再进入下一阶段：

1. 增加槽位提取
2. 增加更细路由分支
3. 接入插件或 HTTP 请求
4. 再考虑子智能体分发

不要跳过“识别稳定”直接进入业务执行链路。

## 10. 上线执行清单

### 10.1 配置清单

上线前逐项确认：

1. 已创建 `对话流型 Beta` 智能体
2. 已接入 Python 前置节点 `intent_mapping_workflow`
3. Python 节点入参固定为：

```json
{
  "action": "get_enabled_intents"
}
```

4. 大模型节点已使用最新版系统提示词
5. 大模型节点已使用最新版用户提示词
6. 大模型节点已显式映射输出字段：

```text
need_clarify
clarify_question
reason_text
primary_intent_code
execution_plan
```

7. 选择器只按 `need_clarify` 分流
8. `execution_plan` 已显式映射并可被下游直接消费

### 10.2 联调清单

联调时逐项确认：

1. Python 前置节点能稳定返回 `intents`
2. 大模型节点能稳定输出合法 JSON
3. 输出的 `intent_code` 必须存在于 `intents`
4. 输出的 `target_agent`、`collaboration_mode` 必须与前置配置一致
5. 模糊输入能正确走 `need_clarify=true`
6. 多意图输入能输出一个主意图和合理的 `execution_plan.root`
7. `execution_plan` 能稳定表达哪些节点串行、哪些节点并行，以及总体执行顺序
8. 嵌套场景下 `execution_plan.root` 能继续向下展开，不混用旧版阶段结构

### 10.3 回归清单

上线前至少回归以下样例：

- TC001
- TC005
- TC010
- TC014
- TC019
- TC022
- TC026
- TC027
- TC030

### 10.4 验收清单

验收通过标准：

- 主意图识别准确率 >= 90%
- 高频场景识别稳定
- 模糊输入优先澄清
- 多意图输入主次关系基本正确
- `execution_plan` 结构稳定，且与主意图、协作顺序一致
- 递归树结构可被下游稳定解析，支持串行/并行嵌套
- 无明显越权输出，不能出现 `intents` 之外的 `intent_code`
- 无明显格式错误，输出 JSON 可被平台稳定解析

### 10.5 上线后观察项

上线后重点观察：

1. 高频误判 Top case
2. 澄清触发率是否过高
3. 多意图输入中 `execution_plan.root` 是否稳定
4. `execution_plan` 是否出现顺序错乱
5. 是否出现未定义 `intent_code`
6. 是否出现主意图和 `target_agent` 不一致

## 11. 长期记忆提示词

如果智能体开启长期记忆，推荐将“记忆抽取”和“记忆更新”分成两段独立提示词配置。

### 11.1 记忆抽取 Prompt

```text
请从用户与助手的对话中提取适合写入长期记忆的用户信息，用于后续商品推荐、知识问答和交易辅助。

提取目标：
只提取对未来多轮对话仍然有价值、且相对稳定的用户信息。

优先提取以下信息：
1. 长期口味偏好
2. 饮食禁忌、忌口、过敏信息
3. 常购品类或明确偏好的商品类型
4. 明确不喜欢的商品特征
5. 价格带偏好、性价比偏好
6. 送礼对象和送礼场景偏好
7. 包装、规格、份量偏好
8. 生活方式相关的稳定消费偏好
9. 用户明确表达的长期需求或固定习惯

不要提取以下信息：
1. 一次性的下单请求
2. 当前会话里的临时任务
3. 当次推荐结果本身
4. 订单号、地址、手机号等敏感信息
5. 模糊、不确定、未确认的信息
6. 助手推测出来的信息
7. 短期情绪和临时表达
8. 明显只在本轮有效的上下文

提取规则：
1. 只记录用户明确表达的信息，不要猜测
2. 只有相对稳定、后续可复用的信息才写入记忆
3. 如果同一信息已经表达过，保留更完整、更明确的版本
4. 如果信息不够明确，不要写入
5. 没有适合长期记忆的信息时，返回空

输出要求：
1. 每条记忆都写成简短、明确、可复用的陈述句
2. 不要添加解释
3. 不要拼接成长段
4. 一条记忆只表达一个事实

示例：
- 用户喜欢辣味零食
- 用户不吃坚果
- 用户更关注性价比
- 用户送礼偏向长辈场景
- 用户偏好小包装零食
- 用户喜欢低糖产品
```

### 11.2 记忆更新 Prompt

```text
你负责维护用户的长期记忆。请根据当前对话中提取到的新记忆，与已有长期记忆进行更新、合并、保留或删除。

目标：
保证长期记忆始终准确、简洁、稳定、可复用，适用于后续商品推荐、知识问答和交易辅助。

更新原则：
1. 新信息明确且更具体时，优先用新信息覆盖旧信息
2. 新信息与旧信息不冲突时，可以保留并补充
3. 新信息与旧信息明显冲突时，保留更新后的最新明确信息
4. 模糊、一次性、临时性信息不能覆盖稳定长期记忆
5. 助手推断的信息不能写入或覆盖记忆
6. 如果当前对话没有新增稳定信息，则不更新记忆
7. 只有当用户明确表达长期偏好已经改变时，才允许新记忆覆盖旧记忆；短期状态、临时想法、一次性需求不能替换长期记忆

适合保留的记忆：
1. 长期口味偏好
2. 稳定忌口、过敏、禁忌
3. 常购品类
4. 稳定价格偏好
5. 稳定送礼偏好
6. 稳定包装规格偏好
7. 明确不喜欢的商品特征
8. 稳定生活场景偏好

删除或替换原则：
1. 如果用户明确否定旧偏好，应删除或替换旧偏好
2. 如果旧记忆过于宽泛，新记忆更具体，应保留更具体版本
3. 如果旧记忆和新记忆只是表达不同但语义一致，应合并去重
4. 如果某条信息明显只属于本次会话，不要保留
5. 敏感信息、一次性交易信息不要进入长期记忆

冲突处理示例：
- 旧记忆：用户喜欢辣味零食
- 新信息：最近不想吃辣了，更想吃清淡的
处理原则：如果只是短期表达，不直接覆盖；如果用户明确表达长期偏好变化，才替换

- 旧记忆：用户更关注性价比
- 新信息：买礼盒时更看重包装体面
处理原则：两者不冲突，可以并存，但要区分场景

- 旧记忆：用户喜欢大包装
- 新信息：平时自己吃更喜欢小包装
处理原则：按场景拆分，保留更精细的记忆

输出要求：
1. 只输出最终应保留的记忆结果
2. 每条记忆必须是简短、明确、可复用的陈述句
3. 一条记忆只表达一个事实
4. 去重后输出
5. 没有有效更新时返回空
```
