# HiAgent意图映射文档

本文严格依据 `docs/来伊份 AI 导购助手 HiAgent 架构文档 V1.1.pdf` 中以下章节整理：

- `1.3 Agent 与业务中心映射`
- `2.0 意图关键词`
- `2.3 HiAgent 智能体协作关系图 / 协作模式`

目标不是重新发明一套意图体系，而是把文档中已经定义的智能体职责、意图入口和协作方式收敛成可执行的意图映射口径。

## 1. 1.3 Agent 与业务中心映射

根据架构文档，当前智能体与业务中心映射如下。

| Agent | 核心职责 | 交互业务中心 |
|------|----------|-------------|
| 意图Agent | 意图识别、槽位补全、统筹协调 | 调度所有业务中心 |
| 交易Agent | 加购、删改商品、数量调整；下单、支付、价格计算；订单查询、物流查询 | 交易中心、商品中心、支付中心、营销中心、订单中心、履约中心 |
| 问答Agent | FAQ问答、商品知识 | 商品中心、营销中心、用户中心 |
| 商推Agent | 商品推荐、场景推荐 | 商品中心、营销中心、用户中心 |
| 用户Agent | 用户画像、偏好记忆 | 用户中心 |
| 闲聊Agent | 闲聊回复 | 豆包大模型 |

从这一章可以直接得到意图映射约束：

1. `意图Agent` 只负责识别、补全、分发，不负责承接最终业务执行
2. `交易Agent` 是交易域主承接方，覆盖购物车、下单、订单、物流
3. `商推Agent` 只承接推荐类需求
4. `问答Agent` 只承接 FAQ 与商品知识类需求
5. `用户Agent` 只承接用户画像与偏好类需求
6. `闲聊Agent` 只承接闲聊回复

## 4. 2.3 HiAgent 智能体协作关系图

根据架构文档 2.3，协作模式分为 3 类：

| 协作模式 | 说明 | 示例场景 |
|---------|------|---------|
| 串行协作 | 前一个 Agent 完成后再调用下一个 | 选商品 -> 加购 -> 下单 |
| 并行协作 | 多个 Agent 同时处理不同任务 | 推荐商品 + 查询用户画像 |
| 嵌套协作 | Agent 内部再调用其他 Agent | 推荐商品 -> 商品详情(问答Agent) |

文档给出的协作流程说明为：

1. 用户发送自然语言请求
2. 意图Agent 接收请求，进行意图识别和槽位补全
3. 意图Agent 根据识别结果，分发给对应业务Agent
4. 业务Agent 调用相关业务中心完成具体功能
5. 业务Agent 返回结果给意图Agent
6. 意图Agent 整合结果，生成最终响应返回给用户

这意味着当前意图映射文档必须遵守两个原则：

1. `意图识别` 与 `业务执行` 分层
2. `意图映射` 只决定由谁承接，不在映射层提前实现业务编排

## 5. 协作模式下的映射规则

### 5.1 串行协作

适用于：

- 商品推荐 -> 添加购物车 -> 确认订单 / 立即下单

映射规则：

- 首意图命中 `view_items` 时，先路由商推Agent
- 用户后续明确表达“加入购物车”，再命中 `add_shop_cart`
- 用户后续明确表达“下单/确认订单/立即下单”，再命中 `confirm_order` 或 `place_order_oow`

### 5.2 并行协作

适用于：

- 推荐商品 + 查询用户画像

映射规则：

- 首意图仍由意图Agent 识别为主意图
- 如果场景需要画像增强，作为内部协作触发 `用户Agent`
- 不在首层意图表中把“画像查询”单独暴露给用户

### 5.3 嵌套协作

适用于：

- 商品推荐过程中需要问答Agent 补充商品详情

映射规则：

- 用户首意图仍命中 `view_items`
- 商品详情、商品知识类追问由商推Agent 内部嵌套调用问答Agent
- 不把所有推荐过程中的知识追问都误映射成首层 `knowledge_quiz`

## 6. 固化调用范式

基于当前 5 个业务智能体，实际可固化的工作流嵌套范式并不多，建议收敛为以下 6 种。

### 6.1 范式 A：单智能体直达

适用：

- `交易Agent`
- `问答Agent`
- `闲聊Agent`

流程：

```text
意图识别 -> 主意图 -> 单个目标Agent -> 返回结果
```

示例：

- 查询购物车
- 查询订单
- 知识问答
- 闲聊回复

### 6.2 范式 B：推荐增强

适用：

- `商推Agent + 用户Agent`

流程：

```text
意图识别 -> 商品推荐 -> 商推Agent
                     -> 并行/前置读取 用户Agent 画像
                     -> 汇总后返回推荐结果
```

示例：

- 给老人推荐礼盒
- 推荐低糖零食

说明：

- 首层主意图仍然是 `view_items`
- `用户Agent` 作为增强信息源，不单独暴露成首层意图

### 6.3 范式 C：推荐问答嵌套

适用：

- `商推Agent + 问答Agent`

流程：

```text
意图识别 -> 商品推荐 -> 商推Agent
                     -> 嵌套调用 问答Agent 获取商品知识/配料/适用人群
                     -> 汇总返回
```

示例：

- 推荐适合老人的礼盒，并说明配料
- 推荐零食，并说明热量

说明：

- 首层仍然以 `view_items` 为主
- 商品知识作为嵌套能力，不必单独打断主流程

### 6.4 范式 D：推荐到交易串行

适用：

- `商推Agent -> 交易Agent`

流程：

```text
意图识别 -> 商品推荐 -> 商推Agent
         -> 添加购物车 -> 交易Agent
         -> 确认订单/立即下单 -> 交易Agent
```

示例：

- 推荐点零食，帮我加购物车，再下单

说明：

- 这是最核心的串行范式
- 当前轮建议只执行主意图
- 后续动作保留在 `execution_plan.root` 中，交给后续轮或嵌套工作流继续处理

### 6.5 范式 E：交易域内部串行

适用：

- `交易Agent` 内部多动作

流程：

```text
意图识别 -> 交易Agent
         -> 查询购物车 / 加购 / 删购 / 清空
         -> 确认订单 / 立即下单 / 查询订单 / 取消订单
```

示例：

- 加购物车然后下单
- 先查订单再取消订单

说明：

- 由于这些动作都归交易域，最适合在交易工作流内部嵌套处理
- 首层不需要拆多个业务Agent

### 6.6 范式 F：意图澄清兜底

适用：

- `意图Agent`

流程：

```text
意图识别 -> need_clarify=true -> 澄清 -> 重新识别 -> 路由业务Agent
```

示例：

- 这个
- 查一下
- 都要

说明：

- 当多意图并列但无法判断执行先后时，优先走该范式
- 不要为了“强行执行”而误选主意图

## 7. 工作流嵌套模板表

为便于在 HiAgent 中直接设计工作流，建议把上面的范式进一步固化成以下模板。

| 模板ID | 模板名称 | 触发条件 | 主 Agent | 嵌套 / 协作 Agent | 输入 | 输出 | 首版是否建议上线 |
|--------|----------|----------|----------|-------------------|------|------|------------------|
| T01 | 单智能体直达 | `execution_plan.root` 仅包含 1 个稳定 intent 节点 | 交易Agent / 问答Agent / 闲聊Agent / 商推Agent | 无 | `primary_intent_code` + 用户原始输入 | 单个业务结果 | 是 |
| T02 | 推荐增强 | 主意图为 `view_items`，且推荐需要用户偏好增强 | 商推Agent | 用户Agent | 用户输入 + 用户标签/画像 | 增强后的推荐结果 | 是 |
| T03 | 推荐问答嵌套 | 主意图为 `view_items`，且推荐结果需要补商品知识 | 商推Agent | 问答Agent | 用户输入 + 推荐候选商品 | 推荐结果 + 商品知识说明 | 是 |
| T04 | 推荐到交易串行 | `execution_plan.root` 中同时存在 `view_items` 与交易域后续动作 | 商推Agent | 交易Agent | 推荐诉求 + 后续交易意图 | 推荐结果或后续串行执行上下文 | 否，建议二期 |
| T05 | 交易域内部串行 | `execution_plan.root` 中命中多个交易意图 | 交易Agent | 交易Agent 内部子流程 | 主交易意图 + 其余交易结构 | 交易执行结果 + 后续待执行动作 | 是 |
| T06 | 澄清兜底 | `need_clarify=true` 或多意图先后无法稳定判断 | 意图Agent | 无 | 用户输入 + 澄清问题 | 澄清后的新输入 | 是 |

### 7.1 模板 T01：单智能体直达

适合：

- 查询购物车
- 查询订单
- 知识问答
- 闲聊回复

工作流建议：

```text
意图识别 -> primary_intent_code -> 单目标Agent -> 返回
```

### 7.2 模板 T02：推荐增强

适合：

- 给老人推荐
- 给儿童推荐
- 低糖、低脂、控卡场景推荐

工作流建议：

```text
意图识别(view_items)
  -> 商推Agent
  -> 读取用户Agent画像
  -> 汇总推荐结果
```

说明：

- 推荐首版可做成“商推主流程 + 用户画像子流程”
- `用户Agent` 更适合做增强，不建议在首层暴露独立用户画像意图

### 7.3 模板 T03：推荐问答嵌套

适合：

- 推荐适合老人吃的礼盒，并说明配料
- 推荐零食，并解释热量/成分/适用人群

工作流建议：

```text
意图识别(view_items)
  -> 商推Agent
  -> 问答Agent 补充商品知识
  -> 汇总推荐结果
```

说明：

- 这是最适合首版上线的嵌套范式之一
- 推荐与问答的边界最清晰，收益也最高

### 7.4 模板 T04：推荐到交易串行

适合：

- 推荐点零食，帮我加购物车，再下单
- 推荐礼盒，然后直接买

工作流建议：

```text
意图识别
  -> 主意图优先执行
  -> 其余意图保留在 execution_plan.root
  -> 后续轮或后续嵌套工作流继续执行
```

说明：

- 这是典型多意图串行范式
- 但首版不建议一步做完到底
- 建议首版只落主意图，其余意图保留在 `execution_plan.root`

### 7.5 模板 T05：交易域内部串行

适合：

- 加购物车然后下单
- 查订单后取消订单
- 先看购物车再确认订单

工作流建议：

```text
意图识别
  -> 交易Agent 主流程
  -> 交易Agent 内部根据 execution_plan.root 再决定后续动作
```

说明：

- 因为都属于交易域，这个模板比跨 Agent 串行更适合首版
- 交易动作不需要首层拆成多个业务 Agent

### 7.6 模板 T06：澄清兜底

适合：

- 这个
- 都要
- 查一下
- 又想推荐又想查订单，但表达没有顺序

工作流建议：

```text
意图识别
  -> need_clarify=true
  -> 输出 clarify_question
  -> 用户补充后重新进入意图识别
```

说明：

- 这是多意图场景下最重要的兜底模板
- 当先后顺序不明时，宁可澄清，不要误执行

### 7.7 首版上线建议

首版建议优先上线：

1. `T01` 单智能体直达
2. `T02` 推荐增强
3. `T03` 推荐问答嵌套
4. `T05` 交易域内部串行
5. `T06` 澄清兜底

首版不建议完整打通：

1. `T04` 推荐到交易串行全链路自动执行

原因：

- 跨 Agent 串行链路长
- 风险点多
- 更依赖槽位和上下文完整性
- 更适合在首版识别稳定后再做

## 8. HiAgent 工作流节点清单

下面把 6 个模板进一步翻译成 HiAgent 中可直接搭建的节点清单。

### 8.1 模板 T01：单智能体直达

推荐节点：

```text
Start
-> Python（intent_mapping_workflow）
-> 大模型（意图识别）
-> 选择器（need_clarify）
-> 对应业务节点 / 业务工作流
-> End
```

节点说明：

- `Python`：返回 `intents`
- `大模型`：输出 `primary_intent_code`
- `选择器`：按 `need_clarify` 分流
- `业务节点`：按 `primary_intent_code` 对接单个 Agent

### 8.2 模板 T02：推荐增强

推荐节点：

```text
Start
-> Python（intent_mapping_workflow）
-> 大模型（意图识别）
-> 选择器（need_clarify）
-> 用户画像工作流 / 用户Agent
-> 商推工作流 / 商推Agent
-> 结果整合
-> End
```

节点说明：

- `用户Agent` 先作为增强信息源输出画像
- `商推Agent` 使用用户画像生成推荐结果
- `结果整合` 可为变量赋值节点或消息节点

### 8.3 模板 T03：推荐问答嵌套

推荐节点：

```text
Start
-> Python（intent_mapping_workflow）
-> 大模型（意图识别）
-> 选择器（need_clarify）
-> 商推工作流 / 商推Agent
-> 问答工作流 / 问答Agent
-> 结果整合
-> End
```

节点说明：

- `商推Agent` 先给出候选商品
- `问答Agent` 对候选商品补充知识、配料、适用人群
- `结果整合` 输出统一回答

### 8.4 模板 T04：推荐到交易串行

推荐节点：

```text
Start
-> Python（intent_mapping_workflow）
-> 大模型（意图识别）
-> 选择器（need_clarify）
-> 商推工作流 / 商推Agent
-> 交易工作流 / 交易Agent
-> End
```

节点说明：

- 首版不建议同轮自动全部执行
- 更推荐：

```text
当前轮先执行 primary_intent_code
其余动作保留在 execution_plan.root
后续轮或子工作流继续执行
```

### 8.5 模板 T05：交易域内部串行

推荐节点：

```text
Start
-> Python（intent_mapping_workflow）
-> 大模型（意图识别）
-> 选择器（need_clarify）
-> 交易主工作流
   -> 交易动作选择器
   -> 加购 / 删购 / 清空 / 查单 / 取消单 / 下单
-> End
```

节点说明：

- 首层只路由到 `交易Agent`
- 交易域内部再依据 `primary_intent_code` 和 `execution_plan.root` 决定后续动作
- 这是最适合沉淀成“交易子工作流”的模板

### 8.6 模板 T06：澄清兜底

推荐节点：

```text
Start
-> Python（intent_mapping_workflow）
-> 大模型（意图识别）
-> 选择器（need_clarify）
-> 消息（clarify_question）
-> End
```

节点说明：

- 这是必须存在的公共兜底链路
- 所有模板都应共享这个分支

### 8.7 公共节点规范

所有模板建议统一保留以下公共节点：

1. `Start`
2. `Python（intent_mapping_workflow）`
3. `大模型（意图识别）`
4. `选择器（need_clarify）`
5. `消息（澄清）`
6. `End`

这样可以把差异收敛到业务工作流内部，而不是让首层编排越来越复杂。

## 9. 实施优先级表

为了避免首版编排过重，建议按下面优先级逐步实施。

| 优先级 | 模板ID | 模板名称 | 业务价值 | 实施复杂度 | 主要依赖 | 主要风险 | 首版建议 |
|--------|--------|----------|----------|------------|----------|----------|----------|
| P0 | T06 | 澄清兜底 | 高 | 低 | 意图识别 + 消息节点 | 澄清话术不准导致回流率高 | 必做 |
| P0 | T01 | 单智能体直达 | 高 | 低 | 意图识别 + 单业务工作流 | 误判会直接路由错 Agent | 必做 |
| P1 | T05 | 交易域内部串行 | 高 | 中 | 交易主工作流 | 多交易动作顺序判断错误 | 建议首版做 |
| P1 | T03 | 推荐问答嵌套 | 中高 | 中 | 商推工作流 + 问答工作流 | 推荐与知识结果整合不稳 | 建议首版做 |
| P1 | T02 | 推荐增强 | 中 | 中 | 商推工作流 + 用户画像能力 | 用户画像质量不稳定 | 建议首版做 |
| P2 | T04 | 推荐到交易串行 | 高 | 高 | 商推工作流 + 交易工作流 + 上下文衔接 | 跨 Agent 串行链路长、槽位依赖高 | 二期再做 |

### 9.1 P0：必须先做

#### T06 澄清兜底

原因：

- 所有复杂多意图场景最终都需要兜底
- 没有澄清分支，误判成本最高

#### T01 单智能体直达

原因：

- 高频场景覆盖最大
- 是所有后续复杂模板的基础能力

### 9.2 P1：首版推荐做

#### T05 交易域内部串行

原因：

- 多意图里最常见的一类就是交易域复合动作
- 同域处理比跨 Agent 串行更稳

#### T03 推荐问答嵌套

原因：

- 推荐 + 商品知识是高频复合场景
- 商推和问答边界清晰，容易落地

#### T02 推荐增强

原因：

- 用户画像能提升推荐质量
- 但画像质量和可用性可能不稳定，建议排在 T03 后

### 9.3 P2：二期再做

#### T04 推荐到交易串行

原因：

- 跨 Agent 串行最复杂
- 更依赖上下文、槽位、执行顺序
- 首版先识别准确，再做自动串行更稳

### 9.4 推荐实施顺序

推荐按以下顺序实施：

```text
T06 -> T01 -> T05 -> T03 -> T02 -> T04
```

### 9.5 每阶段验收重点

P0 阶段：

- 澄清是否稳定
- 单意图是否稳定路由

P1 阶段：

- 同域多动作是否顺序合理
- 推荐与问答 / 用户画像的嵌套结果是否稳定

P2 阶段：

- 跨 Agent 串行是否依赖完整上下文
- 后续动作是否真的适合自动执行
